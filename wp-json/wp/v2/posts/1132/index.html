In JavaScript, objects are wonderfully versatile, and arrays are wonderfully fast. But a couple more items sit in the middle of these two; the lesser know Map<\/code> and Set<\/code>.<\/p>\n

Designed to make data access faster and simpler, you\u2019ll be hard-pressed to find a coding challenge site that doesn\u2019t involve at least one of these (typically Map<\/code>) on more than a few of their challenges.<\/p>\n

Map<\/code> and Set<\/code> share some things, so let\u2019s kick things off with Map<\/code>.<\/p>\n

\n
\n <\/a>
\n Map
\n<\/h2>\n

A Map<\/code> is a key-value store (much like an array), that will allow you you to use just about anything you want as a key, maintains insertion order, has a size property, and rejects duplicate keys. Now that\u2019s a lot of features right there, so let\u2019s look at a basic example first:<\/p>\n

Straight away you can begin to see the performance advantages of a Map<\/code>. No more looping through massive data-sets to find what you need, just call Map.get()<\/code> and you\u2019re done! The classic example for a good Map<\/code> is often a phone-book, and that makes sense, because you can imagine how efficient it is to call upon any random contact from a phone-book using a Map<\/code>, which knows each and every entry in there by heart.<\/p>\n

\n
\n <\/a>
\n Keys
\n<\/h3>\n

But Map<\/code> is also incredibly versatile. Unlike arrays, it\u2019ll let you use just about whatever you want as a key, whether it be strings, integers, objects, or even functions!<\/p>\n

\n
\n <\/a>
\n Looping
\n<\/h3>\n

Map<\/code> will also retain their insertion order, which is incredibly important when you want to know what came first. Now of course, Map<\/code> isn\u2019t an array, so you can\u2019t call mapName[0]<\/code> to get the first entry (as 0<\/code> is a valid key for a Map<\/code>!), so you might be wondering why insertion order matters?<\/p>\n

Well, Map<\/code> also come with a variety of methods you\u2019ll find familiar from my post on for loops<\/a>, such as Map.values()<\/code> , Map.keys()<\/code> , Map.entries()<\/code> and the infamous forEach<\/code> , all of which benefit from the original data being in the order it was inserted.<\/p>\n

\n
\n <\/a>
\n Duplication
\n<\/h3>\n

Finally it\u2019s worth pointing out another helpful Map<\/code> function: Map.has()<\/code> . This is a simple utility for checking whether a given key already exists in a Map<\/code>, and can therefore be used to avoid overwriting existing data (which Map<\/code> will absolutely allow!):<\/p>\n

\n
\n <\/a>
\n Set
\n<\/h2>\n

Now that we\u2019ve discussed Map<\/code> , Set<\/code> is actually super-simple. Think of Set<\/code> as being a Map<\/code> without the keys. So all values, no keys. Simple!<\/p>\n

\n
\n <\/a>
\n Duplication
\n<\/h3>\n

The beauty of this is that it\u2019s a very straightforward interface for dumping data that will retain its insertion order. But one really powerful aspect is that it will not allow data to be non-unique. This means that if you have a 10,000 item data set, and you insert it all into a Set<\/code>, what you\u2019ll be left with will all be unique. In addition, Set<\/code> does (like Map<\/code>) implement .has()<\/code> , but this time around you\u2019ll be checking if the Set<\/code> has a value<\/em>, rather than checking if a Map<\/code> has a key<\/em>.<\/p>\n

If you\u2019re suddenly thinking back to the last time you did a coding challenge, and realising this would have been super helpful, don\u2019t worry, we\u2019ve all been there.<\/p>\n

\n
\n <\/a>
\n Looping
\n<\/h3>\n

Just like with Map<\/code>, Set<\/code> has a number of helpful methods for iteration. You can use Set.keys()<\/code> , Set.values()<\/code> and Set.entries()<\/code> , as well as the classic Set.forEach()<\/code> . Though if you\u2019re sitting there thinking \u201cWait, what keys, and what entries?\u201d, you\u2019d be right. \n <\/a>
\n Communal functionality ( .clear() and .delete() )
\n<\/h2>\n

In addition to the communal functionality and similarities already listed, both Map<\/code> and Set<\/code> also implement clear()<\/code> (which will completely empty out the contents) and delete()<\/code> (which will remove a specific entry (by key for Map<\/code> and by value for Set<\/code>)).<\/p>\n

\n
\n <\/a>
\n When to use Map & Set
\n<\/h2>\n

Now that you know how<\/em> to use these wonderful data structures, you might be wondering when<\/em> to use them. Well the simple answer is: for data-sets (as the name suggests).<\/p>\n

For Map<\/code> , the power of .get()<\/code> is truly incredible. Being able to dump massive amounts of data into a Map<\/code> and then be able to call upon the entry you need without looping is incredibly useful, and comes without the overhead and complexities of the classic Object.<\/p>\n

For Set<\/code>, my personal favourite feature (and one I alluded to earlier) is being able to guarantee a unique set of data. This is logic that would normally be difficult and time consuming, but Set<\/code> just\u2026 does it!<\/p>\n
