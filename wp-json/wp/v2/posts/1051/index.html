As vezes precisamos pegar um commit<\/strong> individual que se encontra em uma outra branch, com alguma feature ou corre\u00e7\u00e3o que aconteceu em paralelo, o Cherry pick<\/code> faz exatamente isso, ele junta dois<\/strong> commits em uma linha do tempo, evitando de repente de realizar um Merge<\/code>.<\/p>\n\n

\n Cherry pick
\n<\/h2>\n

Em alguns momentos nos vemos em uma situa\u00e7\u00e3o onde estamos desenvolvendo em uma branch(chamaremos de Branch X), por\u00e9m existe outro integrante do time que est\u00e1 trabalhando em uma task em outra branch (chamaremos de Branch Y), as duas est\u00e3o sendo desenvolvidas juntas. O dev da branch Y encontrou e corrigiu um bug que impacta diretamente na task da branch X.<\/p>\n

<\/a><\/p>\n

Ao inv\u00e9s de ter que realizar um commit e fazer um merge ou qualquer outra gambiarra (N\u00e3o precisa mandar o c\u00f3digo pelo privado<\/em>) pode ser feito um commit da corre\u00e7\u00e3o e o dev da branch X usa o Cherry pick<\/code> para pegar apenas a corre\u00e7\u00e3o<\/strong> e trazer para a sua linha do tempo.<\/p>\n

Lembrando que o cherry pick n\u00e3o \u00e9 bala de prata, as vezes realmente \u00e9 melhor fazer um merge.<\/em><\/p>\n\n

\n
\n <\/a>
\n Cherry pick na pr\u00e1tica
\n<\/h3>\n

Antes de irmos para o Bash<\/em>, \u00e9 interessante esclarecer que \u00e9 necess\u00e1rio ter duas branchs e pelo menos um commit na branch que voc\u00ea vai pegar o commit.<\/p>\n

Vamos come\u00e7ar bem do in\u00edcio, precisamos criar uma branch:
git checkout -b feat\/header<\/code> <\/p>\n

Realizar um commit:
git commit -am \"Feat: create header\"<\/code><\/p>\n

Criar uma nova branch (a que vai receber o commit, no caso da branch feat\/header<\/em>).
git checkout -b feat\/menu-hamburger<\/code><\/p>\n

<\/a> <\/p>\n

Imagine agora que o dev que est\u00e1 desenvolvendo na branch feat\/header<\/em> encontrou um problema e j\u00e1 corrigiu, e a pessoa que est\u00e1 na feat\/menu-hamburger<\/em> precisava ter acesso a essa corre\u00e7\u00e3o.<\/p>\n

Agora precisamos pegar o HASH<\/em> (Identificador) do commit que queremos, para isso vamos na branch que est\u00e1 o commit:
git checkout feat\/header<\/code><\/p>\n

Rodamos um git log para mostrar os commit e seu respectivo HASH:
git log --oneline<\/code><\/p>\n

Voltamos para a branch que ira receber o commit:
git checkout feat\/menu-hamburger<\/code><\/p>\n

E rodamos o t\u00e3o famoso Cherry-pick<\/em>, lembrando de passar o HASH<\/em>:
git cherry-pick 68bbe92<\/code><\/p>\n

<\/a>
Resolva os conflitos, se existirem<\/em><\/p>\n\n

\n Pegar um conjunto de commit
\n<\/h3>\n

Mas e se eu quiser pegar mais de um commit, ser\u00e1 que d\u00e1? SIM!<\/strong>, o Cherry pick<\/code> suporta esse tipo de a\u00e7\u00e3o.
Segue os mesmo passos do anterior, por\u00e9m, inv\u00e9s de pegar apenas um HASH voc\u00ea precisa pegar o \u00faltimo e o primeiro (Calma, eu n\u00e3o digitei errado, o \u00faltimo realmente vem na frente<\/em>) commit que voc\u00ea quer, e ira pegar todos que estiverem dentro dele, a sintaxe \u00e9 extremamente f\u00e1cil:<\/p>\n

Devemos pegar os dois HASH (o ultimo e o primeiro):
Lembrando que o HASH-A<\/code> (o ultimo) deve estar ap\u00f3s o HASH-B<\/code> (o primeiro), ou seja ser mais velho na linha do tempo.<\/em>
git cherry-pick HASH-A HASH-B<\/code><\/p>\n

\n Conflitos:
\n<\/h3>\n

E para pular, \u2014skip:
