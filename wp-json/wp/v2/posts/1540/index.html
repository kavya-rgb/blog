\n<\/p>\n

\n

There is this big debate out in the dev community about collaborating on code \u2014<\/p>\n

\n

Code reviews vs. Pair programming<\/p>\n<\/blockquote>\n

Surely though, there must be more alternatives. For the longest time, I have been wondering how a code collaboration tool, specifically designed for teams at work could look like.<\/p>\n

This is what prompted me to build an alternative. I would love to hear what you think!<\/p>\n

Sturdy<\/a> is a low overhead code collaboration platform for fast moving teams. It is, of course, open-source<\/a>. Here are the key ideas \u2014<\/p>\n

    \n
  • \ud83d\udc68\u200d\ud83d\udcbb High-level operations around developer intent<\/li>\n
  • \ud83d\ude4c Enable early & fast feedback on code<\/li>\n
  • \ud83d\udd0c Compatible with Git and GitHub<\/li>\n<\/ul>\n

    It is specifically built for collaborating on code at work, and aims to remove most of the overhead associated with contributing code in a team setting.<\/p>\n

    \n
    \n <\/a>
    \n \ud83d\ude4b Why build a new platform?
    \n<\/h2>\n

    I think it’s fair to say that the way we code at work is not the same as open-source. The biggest difference is in how closely developers collaborate with one another \u2014 at work, we do standups, plan together, and share knowledge. <\/p>\n

    On top of that…<\/p>\n

    \n

    Waiting for code reviews sucks!<\/p>\n<\/blockquote>\n

    For developers adopting DevOps and Continuous Delivery, shipping small, incremental changes is crucial. At the same time, the default pull request workflow has a fixed overhead per contribution \u2014 pulling code, creating a branch, staging, committing, pushing etc., not to mention getting a review! This is very pronounced when trying to integrate your code multiple times per day.<\/p>\n

    \n
    \n <\/a>
    \n \ud83e\uddd1\u200d\ud83d\udcbb How is Sturdy different from code reviews?
    \n<\/h2>\n

    With Sturdy, you work in the open within your team. This means you can discover and interact with draft code in your team as it is written. Those team drafts are a bit like live pull requests (think Figma or Google Docs, but using your local editor).<\/p>\n

    You can easily get a copy of someone else’s draft code to your computer, explore it in your IDE and edit the code to automatically create suggestions.<\/p>\n

    \n

    But why? <\/p>\n<\/blockquote>\n

    This enables early, proactive feedback. It means you can help each other in a meaningful way, communicate better, and when everything goes well, your code is pre-reviewed.
    Contributing code should not feel like submitting homework!<\/p>\n

    \n
    \n <\/a>
    \n \ud83d\udc81 Sounds a lot like pair programming. How is Sturdy different?
    \n<\/h2>\n

    Pair programming is great for knowledge sharing and moving fast, but the tradeoff is flexibility \u2014 you have to be there in the moment.<\/p>\n

    Some developers describe Sturdy as \u2014 <\/p>\n

    \n

    asynchronous pair programming<\/p>\n<\/blockquote>\n

    This is because, similar to pairing, everybody in the team can see, try, and give suggestions to draft code. The key difference is that feedback exchange can happen whenever it suits individuals, and doesn’t require you to be in a call.<\/p>\n

    \n
    \n <\/a>
    \n \ud83e\udd37 Is this compatible with my existing setup?
    \n<\/h2>\n

    Yes! Sturdy uses Git as a database but not as a protocol. This means preserving compatibility of the output, while gaining flexibility to design new ways of interacting with code. <\/p>\n

    It also has a GitHub bridge integration, which allows you to use it side by side with your existing setup, CI\/CD and other integrations.<\/p>\n

    \n
    \n <\/a>
    \n \ud83d\udcbb Can I try it by myself, without involving my team?
    \n<\/h2>\n

    Yes, absolutely! Your own workflow will be faster for it. It’s not necessary for an entire team to adopt Sturdy to benefit from a more leveraged VCS experience. <\/p>\n

    With the GitHub bridge set up, by default, Sturdy follows changes on the default (main) branch and allows you to quickly create pull requests.<\/p>\n

    \n
